# 机审规则结果评估与复盘（Excel 输入 → 多 sheet 报告 + 颜色标记）

## 项目简介
本项目模拟机审策略业务场景，以新闻领域为例，设计审核规则后观察机器审核效果，如规则准确性、关键词准确性等，根据效果反馈收严或放宽审核规则，使其准确性最大化

说明：本项目的审核规则非正式机审策略，而是用于观察机审准确性与策略迭代的模拟规则


## 功能概览
输入：audit_samples.xlsx（默认 sheet=raw）  
输出：audit_report.xlsx（多 sheet 报告 + cleaned_raw 颜色标记）

核心能力：
- 数据质量检查：缺失/空字符串/非法值/重复 id 等 → 标记 bad_row + bad_reason，并在 cleaned_raw 里对问题单元格上色
- 统计只基于有效行（bad_row=False），并输出“跳过多少行 + 原因分布”
- 覆盖率/分布：country、decision、未识别比例、country×decision 交叉表（count & row%）
- 效果评估：基于 human_check(TRUE/FALSE) 的 overall accuracy + 按 decision/country/rule_hit 分组 accuracy
- 可选：若存在 human_decision 列（删/人审/通过），自动输出混淆矩阵与误删率/漏删率
- Top wrong cases：按 decision 分组各取 Top10，含 why_wrong（保守模板推断）
- 关键词风险：拆分 triggered_keywords，统计 total_hits/bad_hits/bad_rate，高风险词筛选
- 分层分析：长度桶/是否含英文/source 分层样本数与错误率
- 迭代建议：输出 suggestions 表（>=10 条，证据引用统计结果）


## 业务背景
在机审和模型审核场景中，通常会受到规则准确性及关键词的影响：
一方面，风险拦截能力增强，安全性提升；
另一方面，也可能引入误拒、响应变短、用户体验下降等副作用。

本项目抽象了一个典型问题：
模型架构不变，仅调整安全规则 / 拒答策略 / 部分提示词与数据；
业务反馈 v2 上线后投诉率上升；
需要从日志中评估：规则收紧到底带来了哪些具体影响

## 核心问题
- 规则收紧后，v2 的拒答率是否显著上升？
- v2 的用户反馈是否变差？
- 哪些场景退化最明显？
- 输出长度/时延是否变化？
- 是否出现误拒、高延迟、短回答等 bad case 增长？

## 分析框架

### 数据分层维度
| 维度 | 说明 | 业务意义 |
|-----|------|---------|
| request_id | 请求唯一标识 | 样本粒度追踪 |
| is_refusal | 是否拒答 | 衡量策略收紧程度 |
| refusal_reason | 拒答原因码（RISK_KEYWORD/SELF_HARM等） | 定位规则收紧的具体类型 |
| scenario | 对话场景（coding/medical/finance等） | 识别业务敏感场景 |
| user_vote | 用户反馈（点赞/点踩） | 直接用户体验指标 |
| latency_ms | 响应延迟 | 性能体验指标 |
| output_len | 输出长度 | 内容完整性指标 |
| output_text | 模型输出文本 | 支持输出长度等衍生指标 |

### 核心指标
- **安全与体验指标**
  拒答率：is_refusal = 1 的请求占比
  点踩率：user_vote = -1 的请求占比
  短回答占比：输出长度低于阈值的请求占比
  Bad Case率：误拒 / 高延迟 / 短回答等异常样本占比
- **性能指标**
  P50 / P90 延迟：衡量典型时延与尾部时延
  平均输出长度：衡量回答完整性变化  
- **分层分析维度**
  按 scenario 看场景退化
  按 refusal_reason 看规则影响来源
  按 user_vote 看用户体验反馈
  按版本（v1 / v2）做对比分析


## 分析思路

- 本项目采用“总体对比 + 分层定位 + bad case 下钻”的思路：
  Step 1：总体指标对比
  对比 v1 / v2 的整体拒答率、点踩率、输出长度、时延分位数，判断版本升级是否带来显著变化。
  Step 2：按场景拆分
  对 coding / medical / finance / general_qa 等场景分别计算核心指标，识别退化最明显的业务类型。
  Step 3：按拒答原因拆分
  对 RISK_KEYWORD / SELF_HARM / ILLEGAL / POLICY / OTHER 等拒答原因分布进行分析，定位是哪些安全策略引发了变化。
  Step 4：识别 Bad Case
  通过阈值规则识别以下异常样本：
  疑似误拒
  高延迟
  短回答
  用户点踩样本
  Step 5：输出策略优化方向
  结合指标变化和 bad case 类型，判断问题更可能来自：
  规则过严
  场景识别偏差
  拒答链路过长
  输出截断
  性能开销上升
